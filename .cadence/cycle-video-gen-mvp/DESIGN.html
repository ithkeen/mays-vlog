网页版 AI 视频生成 MVP — 技术设计

前端 Vite + React + TS · 后端 Python + FastAPI · 视频生成走 UCloud 星图 ModelVerse kling-v3 · 视频本体转存 UCloud UFile · 单用户单任务串行

整体形态

前端 SPA 在浏览器跑 UI、本地存历史索引、轮询任务状态。后端 FastAPI 是一层薄代理，承担三件不能放前端的事：持有外部凭证、异步轮询 1～5 分钟的视频任务、把成功视频从 ModelVerse 临时 URL 转存到 UFile 并签发可播放的私有 URL。视频本体不进浏览器，前端只存元数据。

关键流程

① 提交任务

  1. 前端校验后 POST /api/tasks
  2. 后端检查 in-flight 锁，无则锁住 + 写 SQLite (status=pending)
  3. 启 BackgroundTask 后立即返回 {id, status: pending}

② 后台编排

  1. 调 ModelVerse /v1/tasks/submit 拿 task_id
  2. 每 10 秒轮询 status，5 分钟硬超时
  3. 成功 → httpx.stream 拉 mp4 → ufile.putstream 转存
  4. 更新 SQLite (status=success/failure)、释放锁

③ 前端感知

  1. 5 秒间隔 GET /api/tasks/{id}
  2. success → 调 /play_url 拿 1 小时签名 URL → 喂给 <video>
  3. 同时把元数据写 IndexedDB（仅成功）
  4. failure → 弹错，不写 IndexedDB

④ 历史回放

  1. 从 IndexedDB 读列表渲染抽屉
  2. 点某项 → GET /api/tasks/{id}/play_url
  3. <video src=...> 播放 / <a download> 下载
  4. 删除 / 重命名走后端，前端同步更新 IndexedDB

非功能性约束

维度约束
安全API Key / UFile 公私钥仅在后端环境变量；前端不出现任何外部凭证
CORSUFile bucket 必须配 GET/HEAD + Range，暴露 Content-Length / Content-Range / Accept-Ranges
请求体上限FastAPI body 上限调到 ≥ 15MB（覆盖 10MB 图 base64）
超时提交 30s · 轮询 30s/次 · 任务硬超时 5 分钟 · 上传 300s 重试 1 次
可观测结构化日志含 task_id / modelverse_task_id / status / duration_ms / request_id
部署MVP 单机：vite 静态 + uvicorn 单进程；锁是进程内锁，不支持多进程

关键决策

  • 选 kling-v3：单 model id 覆盖文生 + 图生，零分支切换；对应"参数不暴露给用户"最自然
  • 选官方 ufile SDK 不选 boto3：项目锁 UCloud 账号体系，可移植性优势不成立；private_url 一行出 URL 比 boto3 三件套清爽
  • 视频转存 UFile 不存临时 URL：ModelVerse 返回的是临时签名 URL，会过期；只存 URL 几天后历史全 404，必须转存
  • SQLite 不全内存：MVP 也要避免开发期重启丢任务；单文件零运维
  • 前端轮询不用 SSE：5 秒一次足够，单用户场景上 SSE 是负担
  • BackgroundTasks 不用 Celery：单进程单任务串行已能 cover；重启丢 in-flight 是已知 trade-off，启动时把残留 running 行扫成 failure
  • 前端不缓存视频字节：避免 IndexedDB 配额问题；后端是单一权威源，删除 / 重命名走后端
  • 失败任务保留 SQLite 行但不进历史：满足"失败不入库"用户感知，又保留排查能力

留到执行时

  • UFile 预签名 URL 过期上限：本设计签 1 小时，若 US3 不允许联调时降到允许的最大值
  • UFile cn-bj 是否支持 HTTPS：若不支持需换 region 或临时降级 HTTP
  • ModelVerse 视频任务实际 P95 耗时：当前用 5 分钟硬超时，联调后调整
  • ModelVerse 单账号视频任务并发上限：本地串行锁约束并发=1，理论不会触发；若 429 出现需加退避
  • <video> 在 UFile 私有 URL 上的 seek / 下载实际表现：CORS 配齐后浏览器实测
  • 残留 running 行处理策略：MVP 选"标记为 failure"；如要支持断点续轮询后续再做
  • 前端 IndexedDB 与后端列表 diff 策略：MVP 以 IndexedDB 为准；如累积误差再切"以后端为准"
  • bucket 名 / region 具体值：取决于用户控制台实际创建，环境变量传入
